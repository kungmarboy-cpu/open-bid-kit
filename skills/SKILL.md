# Open Bid Kit Pro — 全流程AI智能标书制作平台

融合 **OpenBidKit_Yibiao（易标）、BidMaster-Pro、BiaoShu-SKILL、docSword、Smart-Tender、good-autobid** 六大开源项目精华。

## 核心能力矩阵

| 能力 | 来源项目 | 说明 |
|------|----------|------|
| AI标书生成引擎 | OpenBidKit_Yibiao | 招标解析、内容生成、知识库 |
| 21项合规检查 | BidMaster-Pro | 全维度合规审核 |
| 14步自动流水线 | BiaoShu-SKILL | 全自动标书生成流程 |
| 8行业支持+自检测 | BiaoShu-SKILL | 智能识别行业类型 |
| 专业文档排版 | docSword | 章节编号、样式、题注 |
| 多维审查 | Smart-Tender | 财政/风格/政策/文号/完整性 |
| 简洁3步工作流 | good-autobid | 需求→大纲→终稿 |
| 商机监控 | BidMaster-Pro | 自动抓取招投标公告 |
| RAG知识库 | BidMaster-Pro | 语义重排+向量检索 |
| AI去痕 | BiaoShu-SKILL | 降低内容重复率 |
| OCR扫描件解析 | BidMaster-Pro | 扫描版PDF文字提取 |
| 多模型切换 | BidMaster-Pro | 接入DeepSeek/OpenAI/Ollama等 |

```
流程一（简洁3步）：需求文档 → 大纲生成 → 终稿生成
```

## 工作流一：招标解读 (Interpret)

### 1.1 文档解析

支持多种格式解析：
- PDF（包括扫描件OCR识别）
- DOCX / DOC
- TXT
- XLSX (招标表格)
- 图片（扫描件）

### 1.2 关键信息提取

对招标文件的AI智能解析，提取内容包含：

| 提取项 | 说明 |
|--------|------|
| 项目概述 | 项目名称、背景、规模、预算、时间安排、实施内容 |
| 技术评分要求 | 评分项、权重/分值、评分标准、原文位置 |
| 项目信息 | 项目编号、类型、预算、地址（JSON结构化） |
| 甲方信息 | 招标人名称、地址、联系方式 |
| 项目实施要求 | 期限、阶段、交货时间节点 |
| 资质要求 | 投标人资格条件、证书、人员要求 |
| 无效投标与废标项 | 废标风险项分析 |
| 评标方法 | 评分办法、权重分配 |
| 商务要求 | 报价、付款方式、保证金 |
| 演示/答辩要求 | 投标演示、现场答辩要求 |

### 1.3 风险预警

多维度风险评估，标记潜在废标风险点，生成评分矩阵对照表。

## 工作流二：标书生成 (Generate)

### 2.1 行业自动检测

根据招标文件内容自动识别行业类型，当前支持 **8大行业**：

| 行业 | 典型关键词 |
|------|-----------|
| IT/信息化 | 软件、系统集成、云计算、大数据、信息化 |
| 建筑工程 | 建筑、施工、装修、市政、工程 |
| 医疗健康 | 医疗、医院、设备、药品、医疗器械 |
| 教育服务 | 教育、学校、教学、培训、校园 |
| 制造业 | 制造、设备、自动化、智能制造 |
| 物流运输 | 物流、运输、仓储、配送 |
| 咨询服务 | 咨询、管理、审计、法律、评估 |
| 通用行业 | 其他行业（自动创建行业指南） |

> 如遇到新行业，会自动创建行业指南知识库，形成自进化能力。

### 2.2 评分标准提取与归类

区分两类技术评分：

**技术评分项** — 需在技术方案中逐一响应展开编写的具体内容：
- 方案类：技术方案、实施方案、设计方案
- 措施类：质量保证、安全保障、应急预案
- 团队类：项目组织、人员配备、专家团队
- 实施类：实施计划、进度安排、交付管理
- 服务类：售后服务、技术支持、培训方案

**技术评分要求** — 约束评分/解释评分的通用规则：
- 符合性要求、偏离扣分规则、适用范围说明
- 通用评审规则

### 2.3 大纲生成

基于技术评分大类生成完整目录树：

```
1. 技术方案（对应R1：技术方案完整性）
   1.1 需求分析
       1.1.1 项目现状分析
       1.1.2 建设目标分析
       1.1.3 关键技术要求响应
   1.2 总体设计方案
       1.2.1 设计原则
       1.2.2 技术架构
       1.2.3 功能架构
   1.3 详细实施方案
       ...
2. 项目管理（对应R2：项目管理能力）
   2.1 项目组织与管理
   2.2 进度计划与控制
   ...
```

### 2.4 内容生成

逐章节生成标书内容，结合：
- **项目概述**：响应对应的评分标准和要求
- **全局事实**：预先提取的项目名称、招标人、投标人信息、时间节点等
- **知识库**：从RAG知识库检索相关历史方案和案例
- **行业指南**：行业特定的编写规范和内容要点
- **AI配图**：自动生成架构图、流程图、进度表（Mermaid/图表）

输出要求：
1. 内容必须响应对应评分标准和要求
2. 结合项目特点出具具体实施方案
3. 专业、准确，符合行业规范
4. 包含实施方法、步骤、工具、交付物、保障措施
5. 格式规范，使用标题层级、段落和列表

### 2.5 AI去痕处理

对标书内容进行去痕（降重）处理：
- 检查段落与历史内容的重复度
- 同义改写高度相似段落
- 保持专业性和准确性的前提下降低重复率
- 确保不同章节之间的语气和风格统一

### 2.6 字数管理

根据招标文件要求管理标书字数：
- 检测每个章节的字数
- 对比招标文件要求的字数限制
- 对超长章节给出精简建议
- 对不足章节给出扩充方向

## 工作流三：21项合规检查 (Check)

对标书进行全维度合规审核，覆盖 **21项专业检查规则**：

| 编号 | 检查项 | 说明 |
|------|--------|------|
| 1 | 保证金 | 投标保证金金额、提交方式、提交时间是否合规 |
| 2 | 签字盖章 | 法定代表人签字、公司盖章是否齐全 |
| 3 | 有效期 | 投标有效期是否满足招标文件要求 |
| 4 | 格式一致性 | 文档格式是否符合招标文件要求 |
| 5 | 内容一致性 | 前后文数据、金额、日期等信息是否一致 |
| 6 | 重复率检测 | 内容是否过度重复或抄袭 |
| 7 | 价格合理性 | 报价逻辑和竞争性是否合理 |
| 8 | 资质完整性 | 所需资质证书是否齐全有效 |
| 9 | 人员配置 | 项目人员资历、数量是否满足要求 |
| 10 | 业绩要求 | 类似项目业绩是否达标 |
| 11 | 技术参数响应 | 关键技术参数是否全部响应 |
| 12 | 实质性条款 | 招标文件中标注★的实质性条款是否全部响应 |
| 13 | 偏离表 | 商务/技术偏离表是否填写完整 |
| 14 | 授权委托 | 授权委托书是否规范 |
| 15 | 财务报表 | 财务状况资料是否完整 |
| 16 | 社保纳税 | 社保缴纳和纳税记录是否齐全 |
| 17 | 联合体协议 | 联合体协议书（如有）是否规范 |
| 18 | 分包说明 | 分包方案（如有）是否清晰 |
| 19 | 页码目录 | 页码和目录是否完整准确 |
| 20 | 密封要求 | 投标文件密封格式是否合规 |
| 21 | 递交时间 | 投标截止时间是否符合要求 |

### 3.1 合规检查流程

```
Step 1: 从招标文件提取所有硬性要求
Step 2: 逐条在投标文件中查找对应响应
Step 3: 判断每项是否满足（通过/不通过/需确认）
Step 4: 生成检查报告
```

### 3.2 多维审查

参照 Smart-Tender 的多维审查机制：

| 审查维度 | 审查内容 |
|----------|----------|
| 财政审查 | 预算符合性、资金使用合规性 |
| 风格审查 | 文档格式、字体、行距等风格一致性 |
| 政策审查 | 是否符合相关法律法规和政策要求 |
| 文号审查 | 文号格式是否正确、统一 |
| 完整性审查 | 章节是否齐全、附件是否完整 |

### 3.3 检查报告

输出结构化检查报告：
```json
{
  "total_checks": 21,
  "passed": 17,
  "failed": 2,
  "needs_confirmation": 2,
  "items": [
    {
      "check_id": 1,
      "name": "投标保证金",
      "risk": "high",
      "result": "failed",
      "evidence": "投标保函金额为2万元，招标文件要求5万元",
      "suggestion": "将保函金额修改为5万元"
    }
  ],
  "summary": "整体合规率81%，高风险项2项需立即处理"
}
```

### 4.4 排版模板

提供多种排版模板：

## 工作流五：知识库管理 (RAG + 知识沉淀)

### 5.1 知识库架构

```
knowledge-base/
├── 技术方案/           # 各行业技术方案模板
│   ├── IT信息化/
│   ├── 建筑工程/
│   ├── 医疗健康/
│   └── ...
├── 资质文件/           # 公司资质、人员证书
├── 项目案例/           # 历史项目案例
├── 行业规范/           # 行业标准和规范
└── 行业指南/           # 各行业标书编写指南（自进化）
```

### 5.2 语义检索 (RAG)

- 基于向量检索（ChromaDB/FAISS）的语义搜索
- 支持语义重排（cross-encoder re-ranker）
- 企业画像过滤（按公司、行业、项目类型筛选）
- 自动生成检索报告并附带来源片段

### 5.3 自我进化

- 遇到新行业时自动创建行业指南
- 每次标书生成后沉淀经验到知识库
- 行业指南逐步完善，形成企业知识资产

## 工作流七：脚本工具与集成 (Tools & Integrations)

本插件附带三组可直接运行的脚本工具，覆盖CLI操作、浏览器预览和Notion知识库同步。

### 7.1 CLI标书工具 (bid-agent.py)

```bash
python scripts/bid-agent.py extract <file.md>    # 提取招标文件关键信息
python scripts/bid-agent.py preview <file.md>    # 生成HTML预览文件
```

功能说明：
- **extract**: 自动扫描Markdown标书文件，提取章节结构、字数统计
- **preview**: 使用markdown库将标书渲染为带专业排版的HTML预览，可直接在浏览器中打开

### 7.2 浏览器预览服务器 (preview-server.py)

启动本地HTTP服务器，在浏览器中预览和管理标书文档：

```bash
python scripts/preview-server.py --port 8899
```

功能说明：
- 访问 `http://127.0.0.1:8899` 即可看到所有已生成的标书文档列表
- 点击文档名称进入专业排版预览页面
- 支持在Codex内置浏览器中直接打开预览
- API接口 `GET /api/status` 返回插件运行状态

### 7.3 Notion知识库同步 (notion-sync.py)

将标书模板和知识库与Notion数据库双向同步：

```bash
python scripts/notion-sync.py info     # 查看Notion配置状态
python scripts/notion-sync.py pull     # 从Notion拉取模板
python scripts/notion-sync.py push     # 推送本地知识库到Notion
python scripts/notion-sync.py search <query>  # 搜索Notion知识库
```

配置步骤：
1. 前往 https://www.notion.so/profile/integrations 创建Integration
2. 复制Internal Integration Secret填入 `.env` 的 `NOTION_API_KEY`
3. 在Notion中创建数据库并分享给该Integration
4. 将数据库ID填入 `.env` 的 `NOTION_KNOWLEDGEBASE_DATABASE_ID`

### 7.4 环境配置

复制 `.env.example` 为 `.env` 并填入配置：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| AI_PROVIDER | AI服务商 | deepseek |
| AI_API_KEY | API密钥 | - |
| AI_MODEL | 模型名称 | deepseek-chat |
| NOTION_API_KEY | Notion集成密钥 | - |
| NOTION_KNOWLEDGEBASE_DATABASE_ID | Notion数据库ID | - |

### 7.5 GitHub CI集成

插件内包含 `.github/workflows/validate.yml`，提交到GitHub后自动验证：
- 检查 `plugin.json` 结构完整性
- 验证Python脚本语法正确性
- 确保SKILL.md存在
